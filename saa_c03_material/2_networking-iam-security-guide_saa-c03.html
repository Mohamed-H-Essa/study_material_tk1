Section 01

VPC Fundamentals & Subnets

A Virtual Private Cloud (VPC) is your isolated network within AWS. Every VPC spans a single Region and is defined by a CIDR block (e.g., 10.0.0.0/16). Inside the VPC you create subnets, each mapped to a single Availability Zone.

Public vs Private Subnets

The only difference between a public and private subnet is routing. A public subnet has a route to an Internet Gateway (IGW) in its route table, plus instances need a public or Elastic IP. A private subnet has no IGW route — resources here can only reach the internet through a NAT Gateway (or NAT instance) sitting in a public subnet.

Core pattern: Public subnet → route to IGW. Private subnet → route to NAT Gateway (in a public subnet) → NAT Gateway routes to IGW. Resources in private subnets initiate outbound traffic but cannot be reached from the internet.

Security Groups vs NACLs

Security Groups

  • Instance-level (ENI-level)
  • Stateful — return traffic auto-allowed
  • Allow rules only, no deny
  • All rules evaluated before decision
  • Default: deny all inbound, allow all outbound

Network ACLs

  • Subnet-level
  • Stateless — must allow return traffic explicitly
  • Allow and deny rules
  • Rules evaluated in number order, first match wins
  • Default NACL: allow all in/out
If the exam says "block a specific IP address," you need a NACL deny rule. Security groups have no deny rules.

Explanation

C — route tables don't support per-IP blocking. D — Shield is for DDoS protection, not IP-level blocking.
Section 02

VPC Peering vs Transit Gateway

VPC Peering

A VPC peering connection creates a one-to-one, point-to-point link between two VPCs. Traffic stays on the AWS backbone (not the internet). Key constraints: peering is non-transitive — if VPC A peers with VPC B and VPC B peers with VPC C, A cannot reach C through B. CIDRs must not overlap. Peering works across accounts and across regions (inter-region peering).

For two VPCs this is simple. But the number of connections scales as n(n−1)/2. With 5 VPCs you need 10 peering connections. With 10 VPCs you need 45. It becomes unmanageable.

Transit Gateway

AWS Transit Gateway is a regional hub-and-spoke network. You attach VPCs, VPN connections, and Direct Connect gateways to the Transit Gateway, and it routes traffic between all attachments. It supports transitive routing — attach 50 VPCs and they all reach each other through the single hub. It also supports route tables, enabling segmented architectures (e.g., production VPCs can't talk to dev VPCs).

VPC Peering

  • Best for: 2–3 VPCs
  • Non-transitive
  • No bandwidth bottleneck
  • No additional cost per GB
  • Cross-region supported

Transit Gateway

  • Best for: many VPCs (10+)
  • Transitive routing, hub-and-spoke
  • Supports VPN + Direct Connect
  • Route table segmentation
  • Per-attachment + per-GB cost
The exam's trigger words are clear: "two VPCs need to communicate" → VPC peering. "Multiple VPCs" / "hundreds of VPCs" / "hub-and-spoke" / "transitive routing" → Transit Gateway.
Q3 VPC Connectivity
A company has 30 VPCs across three AWS accounts. They need all VPCs to communicate with each other and with their on-premises data center via a single VPN connection. Which solution minimizes operational overhead?
Explanation
B is correct. Transit Gateway acts as a hub — attach all 30 VPCs and the VPN once. Transitive routing handles the rest. A would require 435 peering connections (30×29/2) plus 30 VPN connections — operationally impossible. C — PrivateLink exposes specific services, not full VPC connectivity. D — consolidating into one VPC breaks isolation and is architecturally unsound.
Q4 Peering Constraints
VPC A (10.0.0.0/16) peers with VPC B (10.1.0.0/16). VPC B peers with VPC C (10.2.0.0/16). A resource in VPC A tries to reach a resource in VPC C. What happens?
Explanation
B is correct. VPC peering is strictly non-transitive. A-to-B and B-to-C exist, but A cannot reach C through B. You would need a direct A-to-C peering connection, or a Transit Gateway.
Section 03

Direct Connect & Encryption

AWS Direct Connect

Direct Connect establishes a dedicated, private physical connection between your on-premises data center and AWS. It bypasses the public internet, providing consistent latency, higher throughput (1 Gbps, 10 Gbps, or 100 Gbps), and reduced bandwidth costs for large data transfers. You connect to a Direct Connect location (colocation facility) and establish a cross-connect to your AWS port.

There are two types of virtual interfaces (VIFs): a Private VIF connects to VPCs (via a Virtual Private Gateway or Direct Connect Gateway), while a Public VIF gives access to AWS public services (like S3) without going through the internet.

The Encryption Problem

Here's the critical exam fact: Direct Connect does NOT encrypt data in transit. It is a private, dedicated line — but the traffic flowing over it is unencrypted. If you need encryption (and many compliance frameworks require it), you must layer a Site-to-Site VPN on top of Direct Connect.

Pattern: Direct Connect alone = private but unencrypted. Direct Connect + VPN = private AND encrypted. This is established by running an IPsec VPN tunnel over the Direct Connect connection via a Public VIF.

Direct Connect Gateway

A Direct Connect Gateway allows a single Direct Connect connection to reach VPCs in multiple Regions. Without it, a Direct Connect connection only reaches VPCs in the Region of its Direct Connect location. With it, one connection can fan out globally.

Any time the exam mentions "dedicated connection" + "encryption," the answer is Direct Connect + VPN. If it just says "dedicated, consistent, low-latency," Direct Connect alone is sufficient.
Q5 Direct Connect Encryption
A financial company uses AWS Direct Connect for a private 10 Gbps link to their AWS environment. A compliance audit requires all data in transit to be encrypted. What is the MOST operationally efficient solution?
Explanation
B is correct. Direct Connect does not natively encrypt traffic. The standard approach is to layer an IPsec VPN on top. A is wrong — there is no such console setting. C abandons Direct Connect's latency and bandwidth benefits. D would encrypt data at rest, but doesn't address transport-level encryption for compliance.
Q6 Direct Connect Architecture
A company has a Direct Connect connection in us-east-1 and needs to access VPCs in us-west-2 and eu-west-1 using the same connection. What should they use?
Explanation
B is correct. A Direct Connect Gateway enables a single Direct Connect connection to access VPCs in multiple Regions. You associate the VPGWs from each region's VPCs to the Direct Connect Gateway. A doesn't solve the on-premises connectivity problem. C works but is not cost-effective. D — while Transit Gateway supports Direct Connect, the simplest solution here is a Direct Connect Gateway.
Section 04

Lambda in a VPC

By default, Lambda functions run in an AWS-managed VPC and have internet access but cannot access resources inside your VPC (like RDS databases or ElastiCache clusters). When you configure a Lambda function to run inside your VPC, it gets an ENI (Elastic Network Interface) in the subnets you specify.

The Connectivity Trap

Once a Lambda function is placed in a private subnet, it loses internet access. This is critical: it can now reach your RDS, your ElastiCache, and your other private resources — but it cannot reach the internet (for external API calls, downloading packages, etc.) or even AWS services that have public endpoints (like DynamoDB, S3 via public endpoint, or SES).

To restore internet access: place the Lambda in a private subnet and route outbound traffic through a NAT Gateway in a public subnet — exactly like any other private-subnet resource. Never put Lambda in a public subnet thinking it will get internet access; Lambda functions do not get assigned public IPs, even in public subnets.

Lambda in VPC recipe: Lambda → private subnet → route table points 0.0.0.0/0 to NAT Gateway → NAT Gateway in public subnet → IGW. For AWS services specifically, you can also use VPC endpoints instead of going through the NAT (cheaper and faster).
If the exam says "Lambda function times out when calling DynamoDB" or "Lambda can't reach an external API after being placed in a VPC" — the answer is almost always: add a NAT Gateway (or a VPC endpoint for AWS services).
Q7 Lambda VPC Networking
A developer configures a Lambda function to run inside a VPC to access an RDS PostgreSQL instance. After deployment, the function successfully queries the database but times out when trying to call an external payment API over HTTPS. What should the developer do?
Explanation
B is correct. Lambda in a VPC needs a NAT Gateway for internet access. A won't work — Lambda doesn't support Elastic IPs or public IPs even in public subnets. C — adding an IGW route makes it a public subnet, but Lambda still won't get a public IP. D would restore internet access but break the RDS connection, which is the whole reason it's in the VPC.
Section 05

VPC Endpoints (Gateway & Interface)

VPC endpoints let resources in your VPC access AWS services without traversing the internet, without a NAT Gateway, and without a VPN. Traffic stays entirely on the AWS network. There are two types:

Gateway Endpoint

  • S3 and DynamoDB only
  • Free of charge
  • Added as a target in the route table
  • Does not use an ENI
  • Controlled by endpoint policies

Interface Endpoint (PrivateLink)

  • Supports 100+ AWS services
  • Hourly + per-GB cost
  • Creates an ENI in your subnet
  • Uses Security Groups for access control
  • DNS resolves service to private IP
Gateway endpoints are free and work for S3 and DynamoDB only. If the exam mentions accessing S3 from a private subnet without a NAT Gateway — Gateway Endpoint. For any other service (SQS, SNS, KMS, etc.) — Interface Endpoint.
Q8 VPC Endpoints
An application in a private subnet needs to write objects to S3 without any traffic leaving the AWS network. The solution must have no additional hourly costs. What should you configure?
Explanation
A is correct. Gateway endpoints for S3 are free of charge, keep traffic on the AWS network, and are added as a route table entry. B works technically but incurs hourly charges ($0.01/hr per AZ) and per-GB fees. C — NAT Gateway also has hourly + per-GB costs and routes through the public internet. D is an unnecessary complexity.

IAM Essentials & Instance Profiles

IAM Roles for EC2 (Instance Profiles)

The golden rule: never store AWS access keys on EC2 instances. Instead, assign an IAM role to the instance via an instance profile. The instance profile is a container that delivers temporary credentials to the EC2 instance through the instance metadata service. Applications running on the instance use the AWS SDK, which automatically retrieves and rotates these short-lived credentials.

Instance Profile = IAM Role delivery mechanism for EC2. When you "attach a role to EC2" in the console, AWS creates an instance profile behind the scenes. Via CLI/CloudFormation, you create the instance profile explicitly.

When to Use What

ScenarioUseWhy
EC2 accessing S3Instance Profile (IAM Role)Temporary creds, auto-rotated
Lambda accessing DynamoDBExecution RoleLambda's equivalent of instance profile
Cross-account accessSTS AssumeRoleTemporary creds in target account
On-prem app accessing AWSSTS AssumeRole or FederationNo long-term keys on-prem
CI/CD pipeline (external)OIDC FederationGitHub Actions, GitLab — no stored secrets
If an exam question describes an EC2 instance using hardcoded access keys, the answer is always "replace with an IAM role/instance profile." This is a near-guaranteed question.
Q9 Instance Profiles
A developer has hardcoded AWS access keys in an application config file on an EC2 instance to access an S3 bucket. A security review flags this as a vulnerability. What is the recommended remediation?
Explanation
C is correct. The best practice is to eliminate long-term credentials entirely. An instance profile delivers temporary, automatically rotated credentials via the metadata service. A still has keys on the instance. B is better than hardcoding but still requires retrieving secrets. D is operational overhead and still uses long-term keys.
Q10 IAM Best Practices
An ECS task needs to read messages from an SQS queue and write results to DynamoDB. How should the permissions be configured?
Explanation
B is correct. ECS uses task roles — the equivalent of instance profiles but at the task/container level. This gives each task its own least-privilege permissions. A uses long-term keys — never do this. C gives all tasks on that instance the same permissions, violating least privilege. D is too broad and doesn't support granular per-task control.
Section 07

Secrets Manager vs Parameter Store

AWS Secrets Manager

Secrets Manager is purpose-built for secrets: database credentials, API keys, OAuth tokens. Its killer feature is automatic rotation. You enable rotation and Secrets Manager uses a Lambda function to periodically rotate the secret and update the service (e.g., RDS) automatically. It natively integrates with RDS, Redshift, and DocumentDB for built-in rotation templates.

Systems Manager Parameter Store

Parameter Store is a general-purpose configuration store. It holds parameters as strings, string lists, or secure strings (encrypted with KMS). It does not have built-in automatic rotation. It's cheaper (free tier for standard parameters) and works well for non-secret configuration: feature flags, AMI IDs, license keys, endpoints.

Secrets Manager

  • Built-in automatic rotation
  • Native RDS/Redshift rotation templates
  • $0.40/secret/month + API calls
  • Cross-account sharing
  • Designed for secrets only

Parameter Store

  • No built-in rotation
  • Free tier (standard), $0.05/advanced
  • Hierarchical structure (/app/prod/db)
  • Up to 8 KB (standard) or 8 KB (advanced)
  • Config + secrets (but no rotation)
The exam trigger is the word "rotation." If the question mentions automatic rotation of database credentials → Secrets Manager. If it's general config storage or non-rotating secrets → Parameter Store is fine.
Q11 Secrets Management
A company stores its RDS MySQL database password in a configuration file. A new compliance policy requires the password to be rotated automatically every 60 days without application downtime. What should the solutions architect recommend?
Explanation
B is correct. Secrets Manager has built-in rotation with native RDS templates — you set the rotation interval and it handles everything. A requires building and maintaining custom rotation logic. C is valid for some use cases but IAM DB auth has limitations (max connections, only MySQL/PostgreSQL, token-based). D is manual and error-prone.
Q12 Config vs Secrets
An application needs to store the following: (1) a database connection string with credentials that must auto-rotate, (2) a feature flag toggle value, and (3) the latest approved AMI ID for deployments. What combination provides the MOST cost-effective and operationally efficient solution?
Explanation
B is correct. Use the right tool for each job: Secrets Manager for the credentials that need rotation; Parameter Store (free tier) for the non-secret config values. A works but costs $0.40/secret/month for items that don't need rotation. C — Parameter Store has no built-in rotation. D — DynamoDB is overkill for simple config values.

STS & SAML Federation

AWS Security Token Service (STS)

STS issues temporary security credentials (access key, secret key, session token) that expire after a configurable period. The core API calls you need to know:

API CallUse CaseWho Calls It
AssumeRoleCross-account access, role switchingIAM users/roles in any account
AssumeRoleWithSAMLEnterprise SSO via SAML 2.0 IdPOn-prem users via SAML assertion
AssumeRoleWithWebIdentityWeb/mobile identity federationUsers authenticated by Google, Facebook, Cognito
GetSessionTokenMFA-protected API accessIAM users needing temp creds with MFA
GetFederationTokenBroker-based custom federationFederated users via custom identity broker

SAML 2.0 Federation (On-Prem to AWS)

The typical enterprise scenario: a company has an on-premises Active Directory and wants employees to access AWS without creating individual IAM users. The flow works like this: the user authenticates with the on-prem IdP (Active Directory + ADFS), the IdP sends a SAML assertion to AWS STS, STS validates the assertion and returns temporary credentials via AssumeRoleWithSAML. The user now has time-limited access to AWS.

SAML federation flow: User → authenticate with corporate IdP → receive SAML assertion → call STS AssumeRoleWithSAML → receive temporary AWS credentials → access AWS resources. No IAM users needed for corporate employees.

AWS IAM Identity Center (SSO)

For new implementations, AWS recommends IAM Identity Center (formerly AWS SSO) over direct SAML federation. It provides a centralized portal for accessing multiple AWS accounts and business applications. It integrates with Active Directory, Okta, and other IdPs, and supports permission sets for managing access across accounts in AWS Organizations.

If the question says "on-premises Active Directory" + "SAML" → STS AssumeRoleWithSAML. If it says "centralized access to multiple AWS accounts" → IAM Identity Center. If it says "mobile app login with Google/Facebook" → Cognito (which calls AssumeRoleWithWebIdentity behind the scenes).
Q13 SAML Federation
A company with 5,000 employees uses on-premises Active Directory for identity management. They want employees to access AWS resources using their existing corporate credentials without creating IAM users. The company uses ADFS as their SAML 2.0 identity provider. What should the solutions architect configure?
Explanation
B is correct. SAML 2.0 federation with ADFS is the standard enterprise pattern. ADFS issues SAML assertions, and STS AssumeRoleWithSAML exchanges them for temporary credentials. No IAM users needed. A — creating 5,000 IAM users violates scalability best practices and has a default limit of 5,000. C — Cognito is for web/mobile app users, not enterprise SSO. D — GetSessionToken is for MFA, not SAML federation.
Q14 Cross-Account Access
Account A (Development) needs to access an S3 bucket in Account B (Production). The access must use temporary credentials and follow least privilege. What is the correct approach?
Explanation
B is correct. Cross-account access via STS AssumeRole is the standard pattern: create a role in the target account, trust the source account, and assume it for temporary credentials. A uses long-term keys. C grants root-level access — not least privilege. D — VPC peering doesn't solve IAM-level access control.
Section 09

WAF, Shield & DDoS Protection

AWS WAF (Web Application Firewall)

WAF operates at Layer 7 (application layer) and inspects HTTP/HTTPS requests. It protects against application-layer attacks: SQL injection (SQLi), cross-site scripting (XSS), HTTP flooding, and bot traffic. You define Web ACLs (Access Control Lists) with rules that inspect request headers, body, URI, query strings, and IP addresses.

WAF deploys in front of CloudFront, ALB, API Gateway, and AppSync. It does NOT work with NLB (Layer 4) or Classic Load Balancers. You can use AWS Managed Rules (pre-built rulesets) or create custom rules. Rate-based rules can block IPs exceeding a request threshold.

AWS Shield

Shield provides DDoS protection at the network and transport layers (Layer 3/4). There are two tiers:

Shield Standard

  • Free, automatic for all AWS accounts
  • Protects against common L3/L4 attacks
  • SYN floods, UDP reflection, etc.
  • Always-on detection

Shield Advanced

  • $3,000/month + data transfer
  • 24/7 DDoS Response Team (DRT)
  • Cost protection (DDoS cost refunds)
  • Advanced metrics, near real-time visibility
  • Works with CloudFront, ALB, EIP, Global Accelerator
WAF vs Shield decision: WAF = application-layer attacks (SQLi, XSS, HTTP floods, bot mitigation). Shield = volumetric DDoS attacks (network/transport layer). They complement each other — use both together. Shield Advanced includes WAF at no additional cost.
If the exam mentions "SQL injection" or "XSS" → WAF. If it mentions "DDoS" → Shield. If it mentions both → Shield Advanced (includes WAF). A common distractor is putting WAF in front of an NLB — remember, WAF only works with CloudFront, ALB, API Gateway, and AppSync.
Q15 WAF Deployment
A web application behind an Application Load Balancer is experiencing SQL injection attacks. The security team needs to block these attacks while allowing legitimate traffic. What should they implement?
Explanation
B is correct. WAF inspects HTTP requests at Layer 7 and has built-in SQL injection detection rules. Attach a Web ACL to the ALB with SQLi rules. A — Shield is for DDoS, not application-layer attacks. C — NACLs operate at Layer 3/4 and cannot inspect HTTP payloads. D — GuardDuty detects threats but does not block them inline.
Q16 DDoS Protection
A company serves a global application through CloudFront and ALB. They want protection against large-scale DDoS attacks, access to a 24/7 response team during attacks, and cost protection so they don't pay for DDoS-induced scaling. Which solution meets ALL requirements?
Explanation
B is correct. Only Shield Advanced provides all three: advanced DDoS protection, 24/7 DDoS Response Team (DRT), and cost protection (AWS credits for DDoS-induced scaling costs). A — Shield Standard is free but has no DRT or cost protection. C — GuardDuty is detection only. D — WAF handles L7 but not volumetric L3/L4 DDoS.
Section 10

GuardDuty, Inspector & Security Hub

Amazon GuardDuty — Threat Detection

GuardDuty is an intelligent threat detection service. It continuously analyzes data sources — VPC Flow Logs, CloudTrail management/data events, DNS logs, and EKS audit logs — using machine learning to identify malicious activity. It detects things like: cryptocurrency mining on your instances, compromised EC2 instances communicating with known C&C servers, unusual API calls indicating credential theft, and S3 bucket enumeration by unauthorized actors.

GuardDuty is detection only — it generates findings but does not block or remediate. You integrate it with EventBridge to trigger automated responses (Lambda, SNS, Step Functions). One-click enablement, no agents to install, no infrastructure to manage.

Amazon Inspector — Vulnerability Scanning

Inspector performs automated vulnerability assessments on your workloads. It scans EC2 instances, container images in ECR, and Lambda functions for software vulnerabilities (CVEs) and unintended network exposure. Inspector v2 is agentless for ECR and Lambda, and uses the SSM agent for EC2.

GuardDuty

  • Threat detection (what's happening now)
  • Analyzes logs + network traffic
  • Finds: crypto mining, C&C, data exfil
  • No agents needed
  • Generates findings → EventBridge

Inspector

  • Vulnerability scanning (what could be exploited)
  • Scans: EC2, ECR images, Lambda
  • Finds: CVEs, open ports, missing patches
  • SSM agent for EC2
  • Risk score per finding

AWS Security Hub

Security Hub is the single pane of glass — it aggregates findings from GuardDuty, Inspector, Macie, Firewall Manager, IAM Access Analyzer, and third-party tools into one dashboard. It also runs automated compliance checks against standards like CIS AWS Foundations, PCI DSS, and AWS Foundational Security Best Practices.

GuardDuty = "Is someone attacking me right now?" Inspector = "Am I vulnerable to attack?" Security Hub = "Show me everything in one place." These three are frequently confused in exam questions — learn the distinction.
Q17 Threat Detection
A company's security team discovers that an EC2 instance is communicating with a known cryptocurrency mining command-and-control server. They want to automatically detect such threats across all accounts in their AWS Organization. Which service should they enable?
Explanation
A is correct. GuardDuty analyzes VPC Flow Logs and DNS logs to detect crypto mining C&C communication. It supports Organization-wide deployment with a delegated administrator. B — Inspector scans for vulnerabilities, not active threats. C — Config checks resource configuration compliance, not real-time network threats. D — Detective is for investigation after a finding, not detection.
Q18 Vulnerability Scanning
A DevOps team pushes container images to Amazon ECR and deploys them to ECS Fargate. They need automated scanning of images for known CVEs (Common Vulnerabilities and Exposures) before deployment. What is the MOST efficient approach?
Explanation
B is correct. Inspector v2 natively integrates with ECR and automatically scans container images for CVEs on push and on new CVE publication. It's agentless and requires no custom setup. A — GuardDuty's ECS runtime monitoring detects threats at runtime, not CVEs in images. C — Config checks resource configuration, not image vulnerabilities. D works but is far more operational overhead.

Systems Manager Session Manager

Replace SSH Entirely

AWS Systems Manager Session Manager provides interactive shell access to EC2 instances (and on-premises managed nodes) without opening inbound ports, without SSH keys, and without bastion hosts. It uses the SSM Agent (pre-installed on Amazon Linux 2/2023 and Windows AMIs) and communicates outbound over HTTPS to the Systems Manager service endpoint.

Why Session Manager > SSH

Traditional SSHSession Manager
Requires port 22 open in Security GroupNo inbound ports needed
SSH key management (creation, distribution, rotation)IAM-based access control
Bastion host needed for private instancesDirect access to private subnet instances
Limited audit trailFull session logging to S3/CloudWatch
Key-based authentication onlyIAM policies + MFA support

How It Works

The SSM Agent on the instance initiates an outbound HTTPS connection to the Systems Manager endpoint. When a user starts a session (from the console, CLI, or SDK), traffic flows through this existing channel. The instance needs: (1) the SSM Agent installed and running, (2) an IAM instance profile with the AmazonSSMManagedInstanceCore policy, and (3) outbound HTTPS (port 443) access to the SSM endpoints (via NAT Gateway or VPC endpoint).

Session Manager requirements: SSM Agent + IAM instance profile with SSM permissions + outbound HTTPS to SSM endpoint. No inbound ports, no SSH keys, no bastion. Session logs go to S3 and/or CloudWatch Logs for complete audit trails.
If the exam says "access EC2 without opening SSH ports" or "eliminate bastion hosts" or "audit all shell sessions" → Session Manager. It's also the answer when the question mentions "private subnet instances" + "no bastion host."
Q19 Session Manager
A security team mandates that all EC2 instances must: (1) have no inbound SSH ports open, (2) all interactive sessions must be logged for audit, and (3) administrators must authenticate using their IAM credentials, not SSH keys. Which solution meets ALL requirements?
Explanation
C is correct. Session Manager checks every box: no port 22 needed, IAM-based authentication (not SSH keys), and full session logging to S3/CloudWatch. A requires port 22 and SSH keys. B still requires port 22 open, even though keys are temporary. D adds operational complexity and doesn't inherently close port 22.
Q20 Session Manager Architecture
An EC2 instance in a private subnet with no NAT Gateway needs to be managed via Session Manager. The instance has the SSM Agent installed and an appropriate instance profile. Sessions are failing. What is the MOST likely solution?
Explanation
B is correct. Session Manager requires outbound HTTPS connectivity to SSM endpoints. Without a NAT Gateway, the instance can't reach the public SSM endpoints. The solution is to create VPC Interface Endpoints for ssm, ssmmessages, and ec2messages, which allow private connectivity. A — Session Manager doesn't use port 22. C — bastions are what Session Manager replaces. D — EIPs don't help in private subnets without an IGW route.

Explanation
