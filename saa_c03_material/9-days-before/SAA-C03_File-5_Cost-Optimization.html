01 Compute purchasing — the decision tree

Match the commitment to the workload's shape

Four ways to buy the same compute, at wildly different prices. The stem tells you which by describing how steady, how interruptible, and how flexible the workload is.

OptionPick it when…Discount
On-Demandspiky, unpredictable, short, can't commit, can't be interruptednone (baseline)
Savings Planssteady spend you'll commit to for 1–3 yr; want flexibilityup to ~66–72%
Reserved Instancessteady, specific instance, 1–3 yr; or you need the capacity reservationup to ~72%
Spotfault-tolerant, stateless, restartable, flexible on timingup to ~90%

Key fact — Savings Plans & RI flavors

Compute Savings Plan = most flexible: covers any instance family, Region, OS, tenancy — and Fargate and Lambda. EC2 Instance Savings Plan = locks a family in a Region for a bigger discount. Standard RI = biggest RI discount but rigid; Convertible RI = exchangeable for different attributes during the term. For most steady workloads, Savings Plans have replaced RIs because of the flexibility — but RIs still appear when a question stresses a specific instance or a capacity guarantee.

The Spot-vs-Reserved trap

When a workload is interruption-tolerant and restartable — nightly batch, ETL, CI, rendering — the greatest savings come from Spot, not Reserved. The fact that it runs every day makes RI tempting, but "can safely restart if interrupted" is the signal that Spot wins. Protect the fleet by diversifying across several instance types and AZs and using the capacity-optimized (or price-capacity-optimized) allocation strategy so one pool drying up can't sink the job.

Two more: On-Demand Capacity Reservations (ODCR) guarantee capacity in a specific AZ (a launch guarantee, not a discount). Dedicated Hosts give you a physical server for socket/core-based BYOL licensing and compliance.

// field note — buying the leftovers

Spot Instances (launched 2009) are AWS selling its spare capacity at up to ~90% off. The catch is they can be reclaimed with a two-minute warning when AWS needs the hardware back. Early Spot used a live bidding market — you literally bid a price — but around 2017 AWS scrapped the auction for smoother, demand-based pricing, so today it's far more predictable. The mental model that survives: Spot is the bargain bin, and you only shop there for things you can afford to put back.

02 Right-sizing, scheduling & elasticity

The cheapest resource is the one that's off

Before you optimize what you pay per hour, cut the hours and the size. This is where most real-world (and exam) savings actually live.

  • Turn it off — dev/test, internal tools, and batch boxes rarely need to run nights and weekends. Schedule stop/start (Instance Scheduler or scheduled scaling).
  • Right-size — use Compute Optimizer's ML recommendations to shrink over-provisioned EC2, ASGs, EBS, and Lambda based on real utilization.
  • Go ARM — Graviton instances (and Lambda on ARM) give better price-performance for compatible workloads.
  • Scale, don't over-provision — let Auto Scaling follow demand instead of running peak capacity 24/7.
  • Fargate Spot for interruption-tolerant containers; gp3 over gp2 for cheaper EBS with independent IOPS/throughput.
  • Storage tiers — S3 lifecycle to Glacier for known cold data; S3 Intelligent-Tiering when the access pattern is unknown or changing (no retrieval fees, no lifecycle logic to build).

Key fact — Lambda cost levers

A Lambda's memory setting also sets its CPU, so right-sizing memory tunes both speed and cost (AWS Lambda Power Tuning helps find the sweet spot). Switching the function to ARM/Graviton typically cuts cost further with no code changes. Provisioned concurrency adds cost — only use it when cold starts genuinely hurt.

// the math that sells scheduling

A weekday-only environment running 8 a.m.–8 p.m. is on for just 60 of the 168 hours in a week — so a schedule that stops it nights and weekends drops roughly 64% of its compute hours, with zero performance trade-off because nobody's using it anyway. Industry FinOps surveys consistently estimate that around a third of all cloud spend is wasted on idle or oversized resources. On the exam, "used only during business hours" almost always means: schedule it off.

03 Cost governance & tooling

You can't optimize what you can't see

The exam expects you to match the question — analyze, alert, detect, or right-size — to the right billing tool.

NeedTool
Visualize & analyze cost/usage trends, filter by service/account/tag, forecastCost Explorer
Most granular line-item billing data for custom analysisCost & Usage Report (CUR) → S3 → Athena/QuickSight
Alert (or act) when spend/usage crosses a thresholdAWS Budgets
ML-detect unusual spend spikes automaticallyCost Anomaly Detection
Right-sizing recommendations from utilizationCompute Optimizer
Quick checks for idle/underused resourcesTrusted Advisor (cost category)
One bill, aggregated volume discounts, shared SP/RI across many accountsOrganizations / consolidated billing

Key fact — analyze across many accounts

To analyze and break down spend across all accounts in an Organization, use Cost Explorer in the management (payer) account — consolidated billing rolls every linked account's cost into one place, and Cost Explorer lets you group by account, service, or tag. Reach for the CUR only when you need the raw line-item detail to build custom dashboards. And cost allocation tags are what make per-team / per-project breakdowns possible in either tool.

// fun aside

The fastest way to cut a cloud bill isn't a clever architecture — it's deleting something. Unattached EBS volumes, idle load balancers, forgotten dev stacks, an EIP nobody released: Trusted Advisor and Compute Optimizer exist mostly to point at the stuff you forgot you were paying for. "Turn it off and right-size it" beats "buy a discount on waste" every single time — which is also the trap behind the oversized Reserved Instance.
