01 Elastic compute & Auto Scaling

Pick the instance, then let the group babysit it

The exam rarely asks "what is EC2." It asks which scaling behavior, which health check, which placement, which automation — with a "least operational effort" clause that quietly rules out anything you'd hand-build.

Instance families, in one breath

The letter is the workload. T/M = general purpose (T is burstable: cheap baseline, CPU credits for spikes). C = compute-optimized (batch, HPC, gaming servers). R/X/z = memory-optimized (in-memory DBs, big caches). I/D = storage-optimized (local NVMe, big sequential I/O). P/G/Inf/Trn = accelerated (GPU/ML). A trailing g means Graviton (AWS ARM silicon) — usually ~20% cheaper and better performance-per-watt when your software runs on ARM. "Lowest cost, x86 not required" is a Graviton tell.

Auto Scaling groups: the four scaling policies

PolicyWhat it doesStem tell
Target trackingKeep a metric at a number (e.g. avg CPU 50%). AWS does the math."maintain", "keep utilization around" → the default right answer
Step scalingAdd/remove N based on how badly an alarm is breached."add more aggressively as load grows"
ScheduledChange capacity at known clock times."every weekday 12–2pm", "business hours only", "predictable calendar"
PredictiveML forecasts recurring patterns and provisions capacity ahead of demand."cyclical", "recurring daily/weekly", "scale before the spike, not after"

Key fact

Health checks decide replacement. By default an ASG only watches EC2 status checks (is the hypervisor happy). If your app crashes but the instance is "running," the ASG leaves it. Turn on ELB health checks so the ASG replaces any instance the load balancer marks unhealthy. The health-check grace period stops it from killing instances mid-bootstrap.

Other ASG levers the exam loves:

  • Launch templates over launch configurations — templates get all new features and versioning; launch configs are legacy.
  • Lifecycle hooks pause an instance in Pending:Wait (run bootstrap/config before it serves) or Terminating:Wait (drain gracefully).
  • Instance refresh rolls a new AMI/launch-template version across the whole group with a rolling replacement — no rebuild.
  • Warm pools keep pre-initialized instances ready, so a slow 8-minute bootstrap doesn't make scale-out useless.
  • Cooldown prevents the group from thrashing before the last action's effect lands.

// field note — design for failure

After a 2008 data-center incident corrupted its database and stopped DVD shipments for three days, Netflix swore off running its own racks and moved onto AWS. To prove the new design could survive failure, it built Chaos Monkey — a tool that randomly kills production instances during business hours. If a dying instance can take you down, you find out at 2 p.m. on a Tuesday, not 2 a.m. during an outage. An Auto Scaling group of stateless instances across multiple AZs is precisely the architecture that makes Chaos Monkey boring.

02 Elastic Load Balancing & traffic entry

Three load balancers, one global front door

The whole game is matching the layer and the requirement: HTTP routing vs raw speed vs inline appliances vs static-IP-across-Regions.

ServiceLayerReach for it when…Can't do
ALB7 (HTTP/S)path/host routing, redirects, WebSockets, OIDC/Cognito auth, target = EC2 / IP / Lambda / containers, WAFstatic IP; non-HTTP protocols
NLB4 (TCP/UDP/TLS)millions of req/s, ultra-low latency, static / Elastic IP, UDP, preserve source IPpath-based HTTP routing; L7 auth
GWLB3+4insert third-party virtual appliances (firewalls, IDS/IPS) transparently; GENEVE :6081be your app's load balancer
Global Accelerator4 (edge)2 static anycast IPs, route over AWS backbone to the best healthy Region, fast cross-Region failover, TCP/UDPcache content (that's CloudFront)

Classic trap

"Static IP" splits two ways. Static IP within a Region for an L4 service → NLB with Elastic IPs. Static IP that survives a whole-Region failure, or non-HTTP traffic that needs the fastest global path → Global Accelerator. And ALB has no static IP at all — its addresses rotate, so any answer pairing "ALB" with "Elastic IP" is bait.

Key fact — cross-zone load balancing

On an ALB it's always on and free. On an NLB (and GWLB) it's off by default, and turning it on bills inter-AZ data transfer. The exam symptom: an NLB where one AZ's targets are slammed while another sits idle because client DNS pinned them to a single AZ node → enable cross-zone load balancing.

Two more entry-tier reflexes: connection draining / deregistration delay lets in-flight requests finish before a target is pulled (so scale-in and deploys don't drop live requests); WAF attaches only to CloudFront, ALB, API Gateway, AppSync — never an NLB.

// field note — trivia + a cautionary tale

The three load balancers arrived years apart: ALB in 2016, NLB in 2017, GWLB in 2020 — each filling a gap the previous one left open. And on Christmas Eve 2012, an AWS operator accidentally deleted Elastic Load Balancing state data in us-east-1; ELB degraded and knocked Netflix streaming offline for hours on one of the year's busiest nights. The lesson lands in File 4: ELB spreads you across AZs for free, but it's still a Regional service — real DR needs a second Region.

03 Session state, placement & the resilient web tier

Stateless instances are the whole point

Every "users lost their sessions during a deploy / scale-in" question has the same answer family: get the state off the instance. This is also where two of your confident-but-wrong storage misses live, so they're back.

Where session state should live

StorePick it whenWhy not the other
ElastiCache for Redissub-ms latency, needs Multi-AZ failover, replication, persistence, cluster mode, sorted setsMemcached has none of that — no replication, no persistence, no Multi-AZ
DynamoDB (TTL)serverless app (API Gateway + Lambda), single-digit ms, auto-expire sessions, zero servers to runRedis means you run a cache cluster; overkill for pure serverless

Sticky sessions are a smell, not a solution

Session affinity pins a user to one instance, so that instance becomes a single point of failure, load goes lopsided, and scale-in / rolling deploys drop sessions. When a stem describes those exact pains, the fix is externalize state + disable stickiness, not "tune the stickiness."

Placement groups — the 30-second version

  • Cluster — pack instances tight in one AZ for lowest latency / highest throughput (HPC, tightly-coupled). Trade-off: a rack/AZ fault can take the whole group. Single-AZ by nature.
  • Spread — scatter across distinct hardware (max 7 per AZ per group) for a handful of critical instances that must not share a failure domain.
  • Partition — up to 7 partitions per AZ on separate racks, hundreds of instances; for big distributed/replicated systems (HDFS, Cassandra, Kafka).

// field note — small team, huge scale

When Facebook acquired Instagram in April 2012 for about $1 billion, the company had 13 employees and ran entirely on AWS. Tens of millions of users, served by a tiny team — because they leaned on managed, auto-scaling, stateless infrastructure instead of operating servers. That's the entire thesis of this section in one acquisition.
