01 SQS — the buffer that never drops work

Put a queue between the tiers and stop losing requests

SQS is a buffer: a producer drops a message, a consumer pulls it when ready, and the queue absorbs every spike in between. Each message is processed by exactly one consumer — SQS does not fan out.

Standard vs FIFO

QueueOrderingDeliveryThroughputReach for it
Standardbest-effortat-least-once (duplicates possible)nearly unlimiteddefault — make consumers idempotent
FIFOstrict, per message groupexactly-once processing + deduplower (per-group)"must be in order" / "no duplicates"

Key fact — visibility timeout

When a consumer receives a message it's hidden for the visibility timeout (default 30s, max 12h). If the consumer doesn't DeleteMessage before that timer expires, the message becomes visible again and another worker can pick it up — that's your duplicate. Fix: set the timeout longer than your worst-case processing time, or extend it with ChangeMessageVisibility as a heartbeat. On Standard queues, duplicates are possible by design, so idempotent consumers are the real insurance.

The rest of the SQS toolbox:

  • Dead-letter queue (DLQ) — after maxReceiveCount failed attempts, a poison message is moved aside so it stops blocking the queue and you can inspect it.
  • Long polling (WaitTimeSeconds up to 20s) — fewer empty responses, lower cost than short polling.
  • Retention — default 4 days, max 14 days. The buffer survives even if every consumer is down.
  • Delay queues / message timers — postpone delivery up to 15 minutes.
  • Scale consumers on backlog — drive an Auto Scaling group off queue depth (ApproximateNumberOfMessagesVisible) or backlog-per-instance.
  • 256 KB max message — for bigger payloads, store in S3 and send a pointer (SQS Extended Client).

// field note — the oldest trick in the book

SQS is one of AWS's very first services, generally available in 2006 — older than the iPhone. The reason it predates almost everything is cultural: Amazon famously reorganized around the idea (recounted as the early-2000s "API mandate") that every team must expose its data and functionality only through hardened service interfaces. Decoupling wasn't a feature they added later — it was the constraint that made AWS possible at all. A queue between two tiers is that same idea in miniature.

02 SNS & EventBridge — fan-out and event routing

One message, many destinations

Where SQS hands a message to one worker, SNS and EventBridge deliver to many — push-style. The difference between them is filtering richness, target variety, and throughput.

SNS — pub/sub fan-out

Publish once to a topic; every subscription gets its own copy. Subscribers can be SQS, Lambda, HTTP/S, email, SMS, or mobile push. The durable pattern is SNS → multiple SQS queues: each consumer gets its own buffered copy and processes at its own pace, surviving downtime. SNS FIFO adds ordering + dedup (delivering to SQS FIFO queues). Message filtering lets a subscription receive only the subset it cares about.

SNS doesn't remember

Plain SNS is fire-and-forward: if a subscriber is offline when a message is published, it misses it — there's no log to replay from. When a stem says consumers must catch up on missed events without data loss, raw SNS is wrong. Buffer it (SNS → SQS) or, for true replay across many independent consumers, use Kinesis (Section 03).

EventBridge — the serverless event bus

EventBridge routes events by content. Rules match an event pattern against the JSON and send to up to five targets each, across a huge range of AWS services, plus SaaS partner sources and AWS service events on the default bus. It adds a schema registry, archive & replay, a cron-style Scheduler, and Pipes for point-to-point filter-and-enrich without glue code.

Key fact — SNS vs EventBridge

Both fan out. Pick SNS for the highest-throughput, lowest-latency pub/sub and for app-to-person messaging (SMS/email/push). Pick EventBridge when you need rich content filtering, many AWS/SaaS targets, schema, or replay — at the cost of higher latency and lower raw throughput. "Add new consumers without touching the producer" and "route based on what's inside the event" both point to EventBridge.

// field note — the timeline

The integration toolkit grew in layers: SQS (2006) for buffering, SNS (2010) for fan-out, Step Functions (2016) for orchestration, Amazon MQ (2017) for migrating legacy brokers, and EventBridge (2019), which started life as CloudWatch Events before graduating into a full event bus. Each one exists because the previous tool was the wrong shape for a new job — which is exactly the judgment the exam tests.

03 Kinesis, orchestration & the decoupling decision

When the answer is a stream, not a queue

This is the section that fixes your most expensive habit. A queue and a stream look similar and feel interchangeable — they are not. Get the deciding factors right and a whole class of questions opens up.

Kinesis: four services, one family

ServiceIt's for
Data Streamsreal-time, replayable log; many consumers each read every record at their own position; ordering per partition key; high throughput
Data Firehosezero-management delivery of streaming data into S3 / Redshift / OpenSearch / Splunk; near-real-time; no shards, no consumers
Managed Service for Apache Flinkreal-time SQL / Flink analytics computed on a stream
Video Streamsingesting and processing media streams

Key fact — SQS vs SNS vs Kinesis

SQS: a message is consumed once by one worker, then deleted — no replay. SNS: pushed to every subscriber once, not retained. Kinesis Data Streams: an append-only log where each consumer reads the whole stream independently, data is retained (24h default, up to 365 days) so a consumer that fell behind can replay, and the partition key keeps records ordered within a shard. When a question says "multiple independent consumers each need every record," "catch up after downtime," "replay/reprocess," and "ordered per key at high throughput" — that combination is Kinesis, every time.

Orchestration & migration

  • Step Functions — coordinate a multi-step workflow with branching, retries, error handling, parallelism, and human approval as a managed state machine. Standard workflows: long-running (up to a year), exactly-once, audited. Express: high-volume, short (<5 min), cheaper per run. Use it instead of hand-wiring Lambdas together with queues when you need state and control flow.
  • Amazon MQ — managed ActiveMQ / RabbitMQ. The answer when you're migrating an existing app that already speaks JMS/AMQP/MQTT/STOMP and you don't want to rewrite it onto SQS/SNS. Building new on AWS? Use SQS/SNS instead.

// fun aside

There's an old engineering joke: the two hardest problems in distributed systems are (1) exactly-once delivery, (2) ordering, and (1) exactly-once delivery. That's not just a gag — it's why "exactly-once" lives in SQS FIFO and Step Functions Standard, why "at-least-once" is the cheap default everywhere else, and why the exam keeps asking you to make idempotency the safety net. If you can hold "at-least-once → make it idempotent" in your head, half the messaging traps disarm themselves.
