Section 01

EBS Volume Types & Limits

Amazon Elastic Block Store (EBS) provides persistent block-level storage volumes for EC2 instances. Choosing the right volume type is one of the most frequently tested topics on the SAA-C03. You need to memorize the IOPS and throughput limits for each type — the exam expects you to pick the correct volume based on workload requirements.

SSD-Backed Volumes (for transactional workloads)

SSD volumes are designed for workloads where the dominant performance attribute is random I/O operations per second (IOPS). There are two families: General Purpose (gp) and Provisioned IOPS (io).

Max Throughput 250 MiB/s 1,000 MiB/s 1,000 MiB/s 1,000 MiB/s 4,000 MiB/s
Max Volume Size 16 TiB 16 TiB 16 TiB 16 TiB 64 TiB
IOPS/GiB Ratio 3 IOPS/GiB 3,000 baseline 50:1 500:1 1,000:1
Multi-Attach No No Yes Yes Yes
Durability 99.8–99.9% 99.8–99.9% 99.8–99.9% 99.999% 99.999%
gp3 vs gp2: gp3 gives you a baseline of 3,000 IOPS and 125 MiB/s regardless of volume size, and you can independently provision more. gp2 ties IOPS to volume size at 3 IOPS per GiB — so you'd need a 5,334 GiB volume to hit max IOPS. gp3 is almost always the better and cheaper choice.
io2 vs io2 Block Express: io2 Block Express is the highest-performance EBS option. It runs on Nitro-based instances and delivers up to 256,000 IOPS, 4,000 MiB/s throughput, and 64 TiB volume size. Regular io2 caps at 64,000 IOPS. The exam may present a scenario needing >64K IOPS — that's your cue for Block Express.

HDD-Backed Volumes (for sequential workloads)

HDD volumes are optimized for large, sequential I/O — think data warehouses, log processing, and big data. They cannot be boot volumes.

Attribute st1 (Throughput Optimized) sc1 (Cold HDD)
Max Throughput 500 MiB/s 250 MiB/s
Max IOPS 500 250
Use Case Big data, data warehousing, log processing Infrequently accessed cold data, lowest cost
Can be boot volume? No No
If a question mentions "boot volume," eliminate st1 and sc1 immediately. Only SSD volumes (gp2, gp3, io1, io2) can be used as boot volumes.
Q1 EBS Volume Types
A company runs a mission-critical Oracle database on an EC2 instance. The database requires sustained 80,000 IOPS with sub-millisecond latency and 99.999% durability. The current io2 volume cannot meet the IOPS requirement. Which solution meets these requirements?
Explanation
C is correct. io2 Block Express supports up to 256,000 IOPS with sub-millisecond latency and 99.999% durability — the only EBS option exceeding 64,000 IOPS. A is a valid technique but adds operational complexity and doesn't provide the 99.999% durability guarantee on a single logical volume. B is wrong because gp3 maxes out at 16,000 IOPS. D provides high IOPS but instance store is ephemeral — data is lost if the instance stops, which is unacceptable for a mission-critical database.
Q2 EBS Volume Types
A developer is running a small development workload on a t3.micro instance. The application needs a root volume with moderate IOPS and the lowest possible cost. Which EBS volume type should the developer choose?
Explanation
B is correct. gp3 provides 3,000 baseline IOPS at a lower price per GiB than gp2 and is the most cost-effective SSD option. A is far too expensive for a dev workload. C and D are HDD volumes that cannot be used as boot/root volumes.
Q3 EBS Volume Types
A data analytics team processes large sequential datasets nightly. They need maximum throughput at the lowest cost and do not need to boot from this volume. Which EBS volume type is MOST cost-effective?
Explanation
A is correct. st1 is designed exactly for large sequential workloads like analytics and log processing, providing up to 500 MiB/s throughput at a low cost. B would work but costs more. C is designed for random I/O, not sequential. D is cheaper but only provides 250 MiB/s throughput — the question asks for "maximum throughput at lowest cost," and st1 delivers 500 MiB/s, making it the best fit.
Section 02

EBS Snapshots, Encryption & Multi-Attach

EBS Snapshots

Snapshots are point-in-time backups of EBS volumes stored in S3 (managed by AWS — you don't see them in your S3 bucket). They are incremental: only blocks that changed since the last snapshot are stored, saving cost and time. You can create volumes from snapshots across Availability Zones and even copy them across regions.

  • Fast Snapshot Restore (FSR): Eliminates the latency penalty of lazy-loading data from S3 when you create a volume from a snapshot. Useful when you need full volume performance immediately (e.g., booting many instances from a snapshot). It's expensive — you pay per AZ per hour it's enabled.
  • EBS Snapshot Archive: Move rarely-accessed snapshots to an archive tier that is up to 75% cheaper. Retrieval takes 24–72 hours. Good for compliance/retention needs.
  • Recycle Bin: Protects against accidental snapshot deletion. You set retention rules (1 day to 1 year), and deleted snapshots go to the recycle bin instead of being permanently removed.

EBS Encryption

EBS encryption uses AWS KMS keys (AES-256) and is transparent to the instance — the OS sees an unencrypted volume. Key facts for the exam:

  • Encryption is handled at the host level (between EC2 instance and EBS volume), so there is minimal performance impact.
  • You cannot directly encrypt an unencrypted volume. The process is: create a snapshot → copy the snapshot with encryption enabled → create a new encrypted volume from the copied snapshot.
  • Snapshots of encrypted volumes are automatically encrypted.
  • You can set a default encryption at the account/region level so all new volumes are encrypted.

Multi-Attach

Multi-Attach allows a single io1 or io2 volume to be attached to up to 16 Nitro-based instances in the same AZ. This is useful for clustered applications (like Oracle RAC) that manage concurrent write operations. The application must handle I/O fencing — EBS does not coordinate writes between instances.

Multi-Attach only works within a single AZ, only with io1/io2, and only on Nitro instances. If a question says "across AZs" or "gp3 multi-attach," it's a wrong answer.
Q4 EBS Snapshots & Encryption
A solutions architect needs to encrypt an existing unencrypted EBS root volume for a running EC2 instance with minimal downtime. Which approach should the architect use?
Explanation
B is correct. You cannot encrypt an existing unencrypted volume in-place. The documented process is snapshot → encrypted copy → new volume → swap. A is not a real capability — modify-volume changes size/type/IOPS, not encryption. C would work but requires launching an entirely new instance, meaning a new IP, new instance ID — more disruption. D only affects newly created volumes, not existing ones.
Q5 Multi-Attach
A company needs to share a single block storage volume between four EC2 instances to run a clustered database. The instances are spread across two Availability Zones in the same Region. Which solution meets this requirement?
Explanation
C is correct. Multi-Attach is limited to a single AZ. Since the instances are in two AZs, you cannot share one EBS volume across them. A would work if all instances were in the same AZ. B uses EFS, which is a file system (NFS), not block storage — the question specifically says "block storage." D is wrong because gp3 does not support Multi-Attach at all.
Section 03

Instance Store vs EBS

Instance store volumes are physically attached to the host machine. They provide extremely high I/O performance — often millions of IOPS — but with a critical trade-off: the data is ephemeral. If the underlying instance stops, terminates, or the hardware fails, all data is lost.

Instance Store

  • Physically attached disk (NVMe SSD)
  • Very high IOPS (millions)
  • Ephemeral — data lost on stop/terminate
  • Cannot be detached/reattached
  • Free (included with instance)
  • Good for: caches, buffers, temp data

EBS Volume

  • Network-attached storage
  • Up to 256K IOPS (io2 Block Express)
  • Persistent — survives stop/restart
  • Can be detached and moved
  • Pay per provisioned GiB/IOPS
  • Good for: databases, OS, anything persistent
If the question mentions "temporary storage," "scratch space," "cache," or "buffer" and wants the highest performance, think instance store. If it mentions "must persist after stop" or "durable," think EBS. If it says "highest possible IOPS" without durability requirements, instance store wins over any EBS type.
Q6 Instance Store vs EBS
An application processes video transcoding jobs. Each job downloads a large file, processes it (requiring very high disk I/O), and uploads the result to S3. The intermediate files do not need to be kept after upload. The team wants to minimize cost while maximizing I/O performance. Which storage configuration is BEST?
Explanation
A is correct. The data is temporary (uploaded to S3 after processing), so persistence isn't needed. Instance store is included with the instance (no extra cost) and provides the highest possible I/O. B works but is expensive and unnecessary for ephemeral data. C adds latency and complexity. D costs more than instance store for lower performance.
Section 04

EFS — Elastic File System

Amazon EFS is a fully managed, elastic NFS file system that can be mounted on multiple EC2 instances simultaneously across multiple AZs. It automatically grows and shrinks as you add and remove files — no pre-provisioning needed.

Key Characteristics

  • Protocol: NFSv4.1 — works with Linux only (not Windows).
  • Scaling: Scales automatically to petabytes. Supports thousands of concurrent connections.
  • Performance Modes: General Purpose (latency-sensitive, default) and Max I/O (higher throughput, higher latency — for highly parallelized workloads like big data).
  • Throughput Modes: Bursting (scales with file system size), Provisioned/Enhanced (decouple throughput from size), and Elastic (automatically scales throughput up/down based on workload — recommended for spiky workloads).
  • Storage Classes: Standard, Infrequent Access (EFS-IA), One Zone, One Zone-IA. Use lifecycle policies to move files to IA automatically. One Zone stores data in a single AZ at ~47% lower cost.
  • Encryption: At rest (KMS) and in transit (TLS) supported.
EFS is Linux-only. If the question mentions Windows instances or SMB protocol, EFS is eliminated. Use FSx for Windows File Server instead.
EFS is the default answer when you see: "shared file system," "NFS," "multiple EC2 instances need access," "auto-scaling storage," or "Linux workload." The exception is high-performance computing — that's FSx for Lustre territory.

All instances need shared access to the same set of uploaded media files. The access pattern is unpredictable — sometimes very heavy, sometimes nearly idle. Which storage solution is MOST cost-effective?
Explanation
B is correct. EFS with Elastic throughput mode automatically scales to match the unpredictable workload — you only pay for what you use. Adding a lifecycle policy to move infrequently accessed files to EFS-IA further reduces cost. A requires you to pre-provision throughput, which wastes money during idle periods. C is not a recommended production solution — s3fs-fuse has limited POSIX compatibility and performance issues. D is for Windows workloads using SMB, not Linux instances.
Q8 EFS
A solutions architect is designing shared storage for a web application. The application runs on Linux EC2 instances in a single AZ and stores mostly static assets that are read frequently but rarely updated. Minimizing storage cost is the top priority, and high availability across AZs is NOT required. Which solution is MOST cost-effective?

Since multi-AZ HA isn't required and files are rarely updated, this is ideal. B costs more due to multi-AZ replication. C is overkill — Lustre is for HPC, not static web assets. D creates a single point of failure and requires manual NFS server management.
Section 05

FSx Family

Amazon FSx provides fully managed third-party file systems. There are four variants, each purpose-built for different workloads:

FSx for Lustre

A high-performance parallel file system designed for HPC, machine learning, video processing, financial modeling, and any compute-intensive workload that needs massive throughput. Lustre is a Linux-native file system used in the world's largest supercomputers.

  • Throughput up to hundreds of GB/s and millions of IOPS.
  • Native integration with S3: can transparently present S3 objects as files. Changes can be lazily loaded or written back to S3.
  • Supports POSIX-compliant file operations.
  • Two deployment types: Scratch (temporary, highest performance, no replication) and Persistent (long-term, data replicated within single AZ).

FSx for Windows File Server

A fully managed Windows-native file system built on Windows Server with full SMB protocol support.

  • Supports Active Directory integration, ACLs, user quotas, DFS namespaces.
  • Accessible from Windows and Linux instances (via SMB).
  • Supports Multi-AZ deployment for high availability.
  • Use case: any time the question mentions SMB, Windows, Active Directory, or NTFS.

FSx for NetApp ONTAP

Managed NetApp ONTAP file system — supports NFS, SMB, and iSCSI. Great for migrating existing NetApp workloads to AWS. Features include snapshots, cloning, replication, compression, deduplication, and tiering.

FSx for OpenZFS

Managed ZFS file system. Supports NFS protocol, point-in-time snapshots, and data compression. Up to 1,000,000 IOPS with low latency. Used for migrating ZFS workloads or when you need ZFS features (snapshots, clones).

Feature FSx Lustre FSx Windows FSx ONTAP FSx OpenZFS
Protocol POSIX (Lustre client) SMB NFS, SMB, iSCSI NFS
OS Support Linux Windows + Linux Windows + Linux + macOS Linux + macOS
S3 Integration Yes (native) No No No
Multi-AZ No (single AZ) Yes Yes No
Key Use Case HPC, ML, big compute Windows enterprise Multi-protocol, migration ZFS migration

Section 06

EFS vs FSx for Lustre — When to Use Which

This is a critical exam distinction. Both are shared Linux file systems, but they serve very different purposes.

Choose EFS When

  • General-purpose shared file storage
  • Content management, web serving
  • Multi-AZ availability needed
  • Auto-scaling storage size
  • Simple NFS mounts
  • Moderate throughput is sufficient
  • Elastic or bursting throughput

Choose FSx for Lustre When

  • HPC, ML training, genomics
  • Need hundreds of GB/s throughput
  • Sub-millisecond latency required
  • Data lives in S3 (transparent integration)
  • POSIX compliance for compute workloads
  • Financial modeling, video rendering
  • Parallel I/O across many nodes
The decision rule is simple: if the question mentions HPC, machine learning, financial simulations, genomics, video rendering, or phrases like "high throughput" / "compute-intensive" + "shared Linux storage" → FSx for Lustre. For everything else that needs shared Linux storage → EFS.
S3 integration is a strong differentiator. If the question says "data is in S3 and needs to be processed by a compute cluster as a file system," that's FSx for Lustre. EFS has no native S3 integration.

The raw data is stored in Amazon S3 and results must be written back to S3. The processing is a one-time batch job that takes 48 hours. Which storage solution is MOST appropriate?
Explanation
B is correct. FSx for Lustre Scratch deployment is designed for temporary, high-performance workloads — it provides the highest throughput without the cost of data replication. Since this is a one-time batch job, persistence isn't needed, making Scratch ideal. The S3 linkage lets the cluster read data transparently. A cannot match Lustre's throughput. C works but Persistent costs more and isn't needed for a temporary job. D is complex to orchestrate and doesn't provide a shared file system view.
Q12 EFS vs FSx for Lustre
A media company has a fleet of Amazon Linux instances that serve a WordPress site. The instances are in an Auto Scaling group across three AZs. They need shared storage for uploaded images and theme files. Traffic is moderate and unpredictable. Which storage is BEST?
Explanation
A is correct. WordPress needs a POSIX-compatible shared file system for uploads and themes. EFS provides multi-AZ NFS access, and Elastic throughput handles unpredictable traffic patterns. B is overkill — Lustre is for HPC, not web serving. C doesn't provide a POSIX file system that WordPress can write to natively. D is fragile and introduces replication lag.

S3 Storage Classes & Lifecycle

S3 offers multiple storage classes optimized for different access patterns. Understanding when to use each — and how lifecycle rules automate transitions — is heavily tested.

Storage Class Availability Min Duration Retrieval Fee Use Case
S3 Standard 99.99% None None Frequently accessed data
S3 Intelligent-Tiering 99.9% None None Unknown/changing access patterns
S3 Standard-IA 99.9% 30 days Per GB retrieved Infrequent but rapid access
S3 One Zone-IA 99.5% 30 days Per GB retrieved Recreatable infrequent data
S3 Glacier Instant Retrieval 99.9% 90 days Per GB retrieved Quarterly access, millisecond retrieval
S3 Glacier Flexible Retrieval 99.99% 90 days Per GB + per request Archive: mins to hours retrieval
S3 Glacier Deep Archive 99.99% 180 days Per GB + per request Long-term archive: 12-48 hrs retrieval

S3 Lifecycle Rules

Lifecycle rules automate object transitions between storage classes or expiration (deletion). Two types of actions:

  • Transition actions: Move objects to another storage class after a specified number of days (e.g., move to Standard-IA after 30 days, Glacier after 90 days).
  • Expiration actions: Delete objects after a specified period, delete old versions, or clean up incomplete multipart uploads.
S3 Intelligent-Tiering is the "I don't know the access pattern" answer. It automatically moves objects between frequent and infrequent access tiers (and optionally archive tiers) with no retrieval fees and no performance impact. There's a small monthly monitoring fee per object.

S3 Versioning & Replication

  • Versioning must be enabled before setting up replication.
  • Cross-Region Replication (CRR): Replicate objects to a bucket in a different region for compliance, lower latency, or DR.
  • Same-Region Replication (SRR): Replicate within the same region for log aggregation, live replication between production and test accounts.
  • Replication is not retroactive. Only new objects after enabling replication are copied. Use S3 Batch Replication for existing objects.
  • Delete markers are not replicated by default (can be enabled).
Q13 S3 Storage Classes
A healthcare company stores patient imaging files in Amazon S3. Regulations require the data to be retained for 7 years. The images are accessed frequently during the first 30 days, then rarely for the next year, and almost never after that. The company wants to minimize storage costs while meeting retrieval requirements (images must be accessible within 12 hours). Which lifecycle configuration is MOST cost-effective?
Explanation
B is correct. This matches the access pattern perfectly: Standard for the first 30 days of frequent access, Standard-IA for the next ~11 months of rare access (with immediate retrieval when needed), and Glacier Deep Archive for years 2-7 (12-48 hour retrieval meets the 12-hour requirement via Expedited or Standard retrieval). A skips the IA tier — Glacier Flexible Retrieval is more expensive than needed during the "rare but still accessed" year. Q14 S3 Replication
A company enables Cross-Region Replication (CRR) on an existing S3 bucket that already contains 500,000 objects. After enabling CRR, they notice that none of the existing objects appear in the destination bucket. Only newly uploaded objects are replicated. What should the architect do to replicate existing objects?

Section 08

Session State Management

This topic appears repeatedly on the SAA-C03. The core principle is straightforward: always externalize session state. Never rely on sticky sessions (session affinity) because they create single points of failure and prevent true horizontal scaling.

The Problem with Sticky Sessions

When a load balancer uses sticky sessions, it routes all requests from a user to the same instance. This means:

  • If that instance fails, the user's session is lost.
  • Traffic distribution becomes uneven — some instances get overloaded while others sit idle.
  • Auto Scaling becomes less effective because sessions are pinned to specific instances.
  • You cannot do rolling deployments smoothly because killing an instance kills its sessions.

The Solution: External Session Stores

Move session data out of the EC2 instances and into a shared, external data store. The two primary options on the exam are:

ElastiCache (Redis)

  • Sub-millisecond latency
  • In-memory key-value store
  • Supports TTL for auto-expiration
  • Cluster mode for high availability
  • Best for: most web app sessions

DynamoDB

  • Single-digit millisecond latency
  • Fully managed, serverless option
  • TTL feature for auto-expiration
  • Built-in HA across 3 AZs
  • Best for: serverless architectures, when you need persistence beyond cache
Exam pattern: Any time you see "stateful application," "user sessions," "session data," or "sticky sessions causing issues" in a question → the answer involves externalizing sessions to ElastiCache Redis or DynamoDB. The question might also say "make the application stateless" — same answer.
ElastiCache Redis is preferred over Memcached on the exam for session stores because Redis supports replication, persistence, and cluster mode for HA. Memcached is simpler but doesn't persist data and doesn't support Multi-AZ. If the question asks for a "highly available session store," pick Redis.

Other Considerations

  • API Gateway + Lambda: These are inherently stateless. If using a serverless architecture, DynamoDB is the natural session store.
  • Cookies: Storing session data in client-side cookies is an option for small amounts of data, but it has size limits (4KB) and security concerns.
  • S3 is NOT a session store. It's an object store with much higher latency — never pick S3 for session management.
Q15 Session State

During peak traffic, the Auto Scaling group launches new instances, but users continue to be routed to overloaded instances because of session affinity. Some users lose their sessions when instances are terminated during scale-in events. How should the architect redesign the application to resolve BOTH issues?
Explanation
B is correct. Externalizing sessions to ElastiCache Redis and disabling sticky sessions solves both problems: traffic distributes evenly across all instances (including newly launched ones), and session data survives instance termination because it's stored externally. A NLB operates at Layer 4 and doesn't solve the session problem. C EBS Multi-Attach is for block storage in a single AZ and is not designed for session state. D delays scale-in but doesn't fix the root cause — sessions are still lost when instances eventually terminate.

A company is building a new serverless application using Amazon API Gateway and AWS Lambda. The application needs to maintain user session data between requests with single-digit millisecond latency. The data must be automatically expired after 24 hours. Which solution is MOST operationally efficient?

Section 09

AWS Storage Gateway

Storage Gateway is a hybrid storage service that connects on-premises environments to AWS cloud storage. It deploys as a VM or hardware appliance in your data center and presents standard storage interfaces (NFS, SMB, iSCSI, iSCSI VTL) while backing data to S3, S3 Glacier, or EBS.

Three Gateway Types

  • S3 File Gateway: Presents an NFS/SMB interface to on-premises applications. Files are stored as objects in S3, with a local cache for low-latency access to recently used data. Great for extending on-premises file shares to the cloud.
  • FSx File Gateway: Provides local cache access to a fully managed FSx for Windows File Server. Useful when on-premises Windows users need low-latency access to a cloud-based Windows file share with AD integration.
  • Volume Gateway: Presents iSCSI block storage backed by S3. Two modes: Cached volumes (primary data in S3 with local cache) and Stored volumes (primary data on-premises with async backup to S3). Used for hybrid block storage and disaster recovery.
  • Tape Gateway: Presents a Virtual Tape Library (VTL) via iSCSI to backup applications like Veeam, Veritas, etc. Tapes are stored in S3 and can be archived to S3 Glacier or Glacier Deep Archive. Used for replacing physical tape infrastructure.
If the question mentions "on-premises backup software" or "tape backups" and migrating to AWS, the answer is Tape Gateway. If it mentions "on-premises file shares accessing S3," that's S3 File Gateway. If it mentions "on-premises Windows file shares with Active Directory," that's FSx File Gateway.
Q18 Storage Gateway
A manufacturing company uses Veritas NetBackup for on-premises backups with physical tape libraries. They want to eliminate tape infrastructure and move backups to AWS, but their backup administrators are familiar with the existing tape workflow and don't want to retrain. Which solution allows them to migrate with MINIMAL changes to their backup process?
Explanation
B is correct. Tape Gateway presents a standard iSCSI VTL interface to backup software like NetBackup. The admins continue using their existing backup workflows with virtual tapes instead of physical ones — minimal retraining. Virtual tapes are stored in S3 and can be archived to Glacier. A would require reconfiguring the backup process. C uses NFS, not the tape interface that NetBackup expects. D transfers files but doesn't integrate with the tape-based backup workflow.
Q19 Storage Gateway
A company has an on-premises application that writes files to an NFS share. They want to make these files accessible in Amazon S3 for analytics while keeping a low-latency local cache for the on-premises application. Which solution is MOST appropriate?
Explanation
A is correct. S3 File Gateway presents an NFS interface locally (so the app doesn't change), caches frequently accessed files for low latency, and stores all data as S3 objects for analytics. B would sync files but doesn't provide a local cache or transparent NFS integration. C uses EFS, not S3, so files aren't directly accessible in S3 for analytics. D is for optimizing uploads to S3 but doesn't provide a local NFS interface or cache.
Section 10

Data Transfer & Migration

AWS provides several tools for moving data into and within the cloud. The choice depends on data volume, network bandwidth, and whether the transfer is one-time or ongoing.

AWS Snow Family

  • Snowcone: 8 TB (HDD) or 14 TB (SSD) usable storage. Smallest and most portable. Can run EC2 instances and IoT Greengrass. Good for edge computing and small migrations in harsh environments.
  • Snowball Edge Storage Optimized: 80 TB usable storage. For large-scale data transfers when network transfer would take too long. Can run EC2 and Lambda@Edge.
  • Snowball Edge Compute Optimized: 42 TB usable storage + powerful compute. For edge computing use cases with pre-processing before transfer.
  • Snowmobile: Up to 100 PB per Snowmobile (a literal truck). For exabyte-scale migrations.
Rule of thumb: If the estimated network transfer time exceeds 1 week (considering available bandwidth), use Snow Family. If the data is >10 TB and bandwidth is limited, Snowball is likely the answer.

AWS DataSync

An online data transfer service for moving large amounts of data between on-premises storage and AWS (or between AWS services). Uses a software agent on-premises. Supports NFS, SMB, HDFS, and self-managed object storage as sources. Destinations include S3, EFS, and FSx. Can transfer up to 10 Gbps per agent and handles scheduling, integrity checks, encryption, and bandwidth throttling.

AWS Transfer Family

Managed service for file transfers into/out of S3 or EFS using SFTP, FTPS, FTP, and AS2 protocols. Used when external partners or systems need to exchange files using legacy protocols.

DataSync is for large-scale, scheduled, or one-time data migrations over the network. Transfer Family is for ongoing file exchange with partners using SFTP/FTP. Snow Family is for offline transfer when network is too slow.
Q20 Data Transfer
A company needs to migrate 60 TB of data from an on-premises NAS to Amazon S3. They have a 100 Mbps Direct Connect link. The migration must be completed within one week. At 100 Mbps, transferring 60 TB would take approximately 55 days. Which migration strategy should the architect recommend?
Explanation
A is correct. The math is clear: 60 TB over 100 Mbps = ~55 days, which exceeds the 1-week deadline by far. Snowball Edge can be loaded locally and shipped, typically completing a transfer in about 1 week (including shipping). B can't overcome the 100 Mbps bandwidth limit — full utilization still takes 55 days. C still limited by the 100 Mbps pipe. D multiple agents doesn't increase the bandwidth of the Direct Connect link itself.
Q21 Data Transfer
A media company receives daily video uploads from hundreds of external contributors via SFTP. The company wants to migrate this workflow to AWS and store the videos in Amazon S3 for processing, without requiring contributors to change their upload method. Which AWS service should the architect use?
Explanation
B is correct. AWS Transfer Family provides a fully managed SFTP endpoint backed by S3. Contributors continue using their existing SFTP clients — no workflow change required. A would require deploying agents at each contributor's site — impractical for hundreds of external parties. C requires contributors to use HTTP/HTTPS, not SFTP. D is for on-premises integration, not for external contributors.
