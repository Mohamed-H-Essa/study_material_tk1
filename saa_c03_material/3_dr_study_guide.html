1. Disaster Recovery Strategies — The RTO Ladder

AWS defines four DR strategies ordered from cheapest/slowest to most expensive/fastest. The exam WILL test you on picking the right one given RTO/RPO constraints and cost concerns.

The Ladder (memorize this order)

StrategyRTORPOCostKey Idea
Backup & RestoreHoursHours$Just backups (S3, snapshots). Restore from scratch when disaster hits.
Pilot Light10s of minutesMinutes$$Core DB/data replication running. App servers OFF. Turn them on when needed.
Warm StandbyMinutesSeconds-Minutes$$$Scaled-down but FULLY FUNCTIONAL copy running. Scale up on failover.
Multi-Site / Hot StandbyNear-zero (seconds)Near-zero$$$$Full production clone running. Route53 or Global Accelerator for instant failover.
Exam Traps:
  • Pilot Light vs Warm Standby — Pilot Light has servers OFF (only data layer runs). Warm Standby has everything running at reduced capacity. If the question says "minimal running infrastructure" → Pilot Light. If it says "scaled-down version of production" → Warm Standby.
  • "Cost-effective" + "RTO of a few hours" → Backup & Restore.
  • "RTO must be under 1 minute" → Multi-Site Active-Active.
  • RPO of near-zero usually implies synchronous replication → Multi-Site or Aurora Global Database.
  • Route 53 failover routing is a common component in Warm Standby and Multi-Site.

Key Services per Strategy

  • Backup & Restore: AWS Backup, S3 Cross-Region Replication, EBS Snapshots, RDS automated backups, AMIs copied cross-region.
  • Pilot Light: RDS read replica (cross-region), DynamoDB Global Tables, Aurora Global Database (data running; compute off).
  • Warm Standby: Same as Pilot Light + EC2/ECS running at minimum capacity behind an ALB + Route 53 failover.
  • Multi-Site: Full mirrored environment + Route 53 active-active (weighted/latency routing), Global Accelerator, Aurora Global Database with <1s replication lag.

Aurora Global Database

Frequently tested. Provides cross-region replication with <1 second replication lag. The secondary region can be promoted in <1 minute. This is the go-to for "near-zero RPO, RTO under 1 minute for RDS/Aurora" scenarios.

CloudEndure / AWS Elastic Disaster Recovery (DRS)

Block-level continuous replication of on-prem or cloud servers to AWS. Provides RPO of seconds, RTO of minutes. If the question mentions migrating on-prem DR to AWS with low RPO/RTO → think DRS.

Section 1 Quiz — DR Strategies

Q1. A company runs a critical e-commerce application in us-east-1. They need a DR strategy with an RTO of less than 1 minute and near-zero RPO. Cost is not a primary concern. Which strategy should they use?

RTO < 1 min + near-zero RPO + cost not a concern = Multi-Site Active-Active. Aurora Global Database gives <1s replication and <1 min promotion. Pilot Light/Warm Standby can't reliably hit sub-minute RTO. Backup & Restore is hours.

Q2. A startup wants the MOST cost-effective DR solution. They can tolerate an RTO of several hours and RPO of 24 hours. Their application runs on EC2 with an RDS MySQL database. Which approach is best?

"Most cost-effective" + hours of RTO + 24hr RPO = Backup & Restore. Daily snapshots copied cross-region + AMIs stored in S3. No running infrastructure in the DR region. Pilot Light still runs a DB continuously (costs more). Warm Standby and Multi-Site are overkill.

Q3. A company uses a Pilot Light DR strategy. Which components are expected to be running in the DR region BEFORE a disaster occurs?

Pilot Light = data layer (DB) is live and replicating. Compute (app servers) is OFF. On disaster, you spin up app servers using pre-configured AMIs/launch templates. B = Multi-Site, C is backwards (that would be compute on, DB off which isn't a real strategy), D = Backup & Restore.

Q4. A company needs to migrate their on-premises VMware-based DR to AWS. They require continuous replication with RPO in seconds and RTO in minutes. Which AWS service should they use?

AWS DRS (formerly CloudEndure Disaster Recovery) provides continuous block-level replication, RPO of seconds, RTO of minutes. It's purpose-built for this. Migration Hub is an orchestration/tracking tool. DataSync is for data transfer (NFS, S3, etc.), not full server replication. SMS is deprecated.

Q5. An architect is designing a Warm Standby DR strategy. Which of the following BEST describes the DR environment?

Warm Standby = fully functional but at reduced capacity. It handles some traffic or can be scaled up quickly. A = Pilot Light, C = Multi-Site, D = Backup & Restore.

Q6. A company uses Aurora MySQL in us-east-1. They need cross-region DR with the ability to promote a secondary region in under 1 minute. Which feature should they use?

Aurora Global Database provides <1s replication lag and can promote a secondary region in <1 minute. RDS cross-region read replicas take longer to promote. Multi-AZ is same-region only. Automated backups need full restore (hours).

2. SQS — FIFO vs Standard

SQS is heavily tested. You need to know when to pick FIFO vs Standard and the specific limits/behaviors of each.

FeatureStandardFIFO
OrderingBest-effort (no guarantee)Strict FIFO within message group
DeliveryAt-least-once (duplicates possible)Exactly-once processing
ThroughputNearly unlimited300 msg/s (3,000 with batching) without high-throughput; 70,000 msg/s with high-throughput mode
DeduplicationNone built-in5-minute deduplication window (content-based or explicit dedup ID)
Queue NameAny nameMust end in .fifo
Use CaseDecoupling, fan-out, high throughputOrder-sensitive: financial txns, command sequences

Message Group ID (FIFO only)

Messages within the same MessageGroupId are delivered in strict order. Different group IDs can be processed in parallel. Think of it as "ordered lanes." If you have 5 group IDs, you can have 5 consumers processing in parallel — each lane is FIFO within itself.

Deduplication (FIFO only)

Two methods: (1) Content-based deduplication — SQS hashes the body and rejects duplicates within 5 min. (2) Explicit MessageDeduplicationId — you provide a unique ID per message. If the same dedup ID is sent within 5 min, it's discarded.

Key Exam Patterns

  • "Messages must be processed in order" → FIFO queue.
  • "No duplicate processing" → FIFO (exactly-once).
  • "Maximum throughput / high volume" → Standard queue.
  • "Financial transactions in sequence" → FIFO with MessageGroupId per account.
  • "Decouple microservices" + no ordering requirement → Standard.
  • FIFO + SNS: SNS FIFO topic can only send to SQS FIFO queues (not standard). SNS Standard can send to both.

DLQ (Dead-Letter Queue)

Both Standard and FIFO support DLQs. The DLQ must be the same type as the source queue (Standard→Standard, FIFO→FIFO). Messages that fail processing after maxReceiveCount times go to the DLQ. The exam loves pairing DLQ with Lambda error handling.

Visibility Timeout

Default 30 seconds. When a consumer receives a message, it becomes invisible to other consumers for this duration. If the consumer doesn't delete it in time, it reappears. If your Lambda takes 5 minutes → set visibility timeout > 5 minutes. Common exam trap: messages being processed twice → increase visibility timeout.

Long Polling vs Short Polling

WaitTimeSeconds > 0 enables long polling (up to 20s). Reduces empty responses and API calls → saves cost. The exam expects you to know this as a cost optimization for SQS.

Section 2 Quiz — SQS

Q1. A financial services company processes stock trades. Each trade for a given account must be processed in the exact order received, and no trade should be processed more than once. Which SQS configuration is correct?

Order per account + no duplicates = FIFO. Setting MessageGroupId to account ID ensures per-account ordering while allowing parallel processing across accounts. A single group ID (D) would serialize ALL trades, destroying throughput. Standard queues (A, C) don't guarantee ordering. Standard queues don't have built-in deduplication (C is invalid).

Q2. A development team notices their SQS Standard queue consumers are processing some messages twice. The application cannot tolerate duplicate processing. They cannot change the queue type. What should they do?

Standard queues are at-least-once — duplicates are inherent. Content-based deduplication is FIFO-only (A is wrong). Since they can't change queue type, the fix is consumer-side idempotency. A DynamoDB table tracking processed message IDs is the standard pattern. Visibility timeout of 0 (B) would make things worse. Long polling (C) reduces cost but doesn't prevent duplicates.

Q3. An SNS FIFO topic needs to fan out messages to multiple subscribers. Which subscriber types are supported?

SNS FIFO topics can ONLY deliver to SQS FIFO queues. They do not support Lambda, HTTP, email, or any other subscriber type. This is a common exam gotcha. If you need Lambda + ordering, you'd subscribe Lambda to the SQS FIFO queue, not directly to the SNS FIFO topic.

Q4. A company's Lambda function processes messages from an SQS queue. They notice messages appear in the queue again after being processed. The Lambda function takes approximately 3 minutes to complete. The queue's visibility timeout is 30 seconds. What is the MOST likely cause and fix?

Lambda takes 3 min but visibility timeout is 30s. After 30s, SQS assumes the message wasn't processed and makes it visible again → another Lambda invocation picks it up. Fix: increase visibility timeout to at least 6x the Lambda timeout (AWS recommendation) or at minimum longer than processing time. Best practice: set it to 6× the function timeout.

Q5. A solutions architect is optimizing SQS costs. The application polls the queue frequently but most polls return empty. Which change reduces costs the most?

Long polling (WaitTimeSeconds > 0, max 20) makes the ReceiveMessage call wait until a message arrives or the timeout expires. This eliminates empty responses → fewer API calls → lower cost. FIFO queues cost more per API call. Retention period and encryption don't affect polling costs.

Q6. A company uses an SQS FIFO queue. They need to increase throughput beyond 300 messages per second without batching. What should they enable?

SQS FIFO high throughput mode supports up to 70,000 messages/second per queue (per API action). This is the direct answer. There's no "auto scaling" or "provisioned throughput" for SQS. Switching to Standard loses ordering guarantees.

3. DynamoDB — On-Demand vs Provisioned Capacity

On-Demand Mode

  • Pay per request (read/write). No capacity planning needed.
  • Scales instantly to handle any traffic level.
  • More expensive per request than provisioned (roughly 5-7x at steady state).
  • Best for: unpredictable/spiky workloads, new tables with unknown traffic, dev/test environments.
  • You CAN switch between modes (once every 24 hours).

Provisioned Mode

  • You specify Read Capacity Units (RCU) and Write Capacity Units (WCU).
  • 1 RCU = 1 strongly consistent read/s for items up to 4KB, or 2 eventually consistent reads/s for items up to 4KB.
  • 1 WCU = 1 write/s for items up to 1KB.
  • Can enable Auto Scaling to adjust RCU/WCU based on utilization targets.
  • Supports Reserved Capacity (1-year or 3-year commitment) for significant discounts.
  • Best for: predictable workloads, cost optimization on steady traffic.

When the Exam Tests This

  • "Unpredictable traffic spikes" → On-Demand.
  • "Cost optimize a table with steady, predictable traffic" → Provisioned + Auto Scaling (+ Reserved Capacity for even more savings).
  • "New application, unknown traffic patterns" → On-Demand initially, switch to provisioned once patterns emerge.
  • "ProvisionedThroughputExceededException" → Either increase provisioned capacity, enable auto scaling, switch to on-demand, or use DAX for reads.
  • "Burst capacity" → DynamoDB reserves unused capacity for bursts (up to 5 minutes of unused capacity). If you exhaust burst → throttling.

DynamoDB Global Tables

Multi-region, multi-active replication. Requires DynamoDB Streams enabled. Provides sub-second replication across regions. Any region can accept writes. This is the DynamoDB DR strategy — exam pairs this with "multi-region active-active" requirements.

DAX (DynamoDB Accelerator)

In-memory cache for DynamoDB. Microsecond read latency. Sits between your app and DynamoDB — no code changes needed (same API). Use for read-heavy workloads. Does NOT help with writes. If the question mentions "reduce DynamoDB read costs" or "microsecond latency for reads" → DAX.

DynamoDB Streams

Ordered log of item-level changes. Can trigger Lambda functions. Used for: replication (Global Tables uses it), aggregation pipelines, cross-service sync. Retention: 24 hours.

Section 3 Quiz — DynamoDB

Q1. A gaming company launches a new mobile game. They have no idea what traffic will look like — it could be 100 users or 10 million in the first week. They use DynamoDB. Which capacity mode should they start with?

Unknown traffic + potential massive spikes = On-Demand. It scales instantly with no capacity planning. Provisioned with high values wastes money if traffic is low. Auto Scaling has lag and may not react fast enough for sudden viral spikes. Reserved Capacity requires commitment with unknown traffic — risky.

Q2. A company has a DynamoDB table with steady, predictable traffic of 5,000 reads/s and 2,000 writes/s. They want to MINIMIZE costs. Which approach is best?

Steady predictable traffic = Provisioned is cheaper. Adding Auto Scaling handles minor fluctuations. Reserved Capacity (1 or 3 year) gives the biggest discount on top. On-Demand would be 5-7x more expensive at this steady rate. D without Auto Scaling risks throttling during minor spikes.

Q3. An application experiences ProvisionedThroughputExceededException errors during peak hours but the table is idle at night. The team wants an automated solution. What should they do?

Predictable peak/idle pattern = Provisioned + Auto Scaling is cost-optimal. It scales up for peaks and down at night automatically. On-Demand would work but is more expensive for a predictable pattern. DAX only helps reads, and the error could be from writes too. Manual changes (D) don't scale.

Q4. A social media company needs their DynamoDB-backed application to serve users across US, Europe, and Asia with single-digit millisecond reads in each region. Users in any region can write data. Which feature should they use?

Multi-region + writes in any region = Global Tables. It's multi-active (read/write in all regions) with sub-second replication. DAX is single-region cache. DynamoDB doesn't have "cross-region read replicas" as a native feature (B is made up). Streams + Lambda replication is what Global Tables does under the hood, but you shouldn't build it yourself.

Q5. A reporting application reads the same DynamoDB items thousands of times per minute. The data rarely changes. The team wants to reduce DynamoDB read costs and improve latency. What should they add?

Read-heavy, same items, rarely changing = DAX is the purpose-built answer. It's a DynamoDB-native cache with microsecond reads and requires no code changes (same DynamoDB API). ElastiCache works but requires code changes and isn't DynamoDB-specific. CloudFront can't directly cache DynamoDB API calls. Streams + pre-compute is over-engineering.

Q6. A company has a DynamoDB table in on-demand mode. After 6 months, they observe consistent traffic of 1,000 RCU and 500 WCU with minimal variation. A solutions architect recommends switching to provisioned mode. How often can they switch capacity modes?

DynamoDB allows switching between on-demand and provisioned once every 24 hours. Not unlimited, not monthly — 24 hours is the specific limit the exam tests.

4. RDS Proxy for Lambda

The Problem

Lambda functions scale horizontally — 1,000 concurrent invocations means 1,000 separate database connections. Relational databases (RDS, Aurora) have connection limits. At scale, you get TooManyConnections errors, the DB becomes CPU-bound managing connections, and everything falls over.

The Solution: RDS Proxy

RDS Proxy sits between Lambda and RDS/Aurora. It pools and shares database connections. Instead of each Lambda opening its own connection, they all go through the proxy which reuses a smaller pool of connections to the database.

Key Benefits

  • Connection pooling: Multiplexes many Lambda connections onto fewer DB connections.
  • Failover: Reduces Aurora/RDS failover time by up to 66% by keeping connections alive during failover.
  • IAM authentication: Lambda can use IAM roles to authenticate to the proxy instead of storing DB credentials.
  • Secrets Manager integration: Proxy pulls DB credentials from Secrets Manager — no credentials in Lambda code.

Architecture

Lambda → RDS Proxy (in VPC) → RDS/Aurora. The proxy MUST be in the same VPC as the database. Lambda must also be configured with VPC access (ENI in the VPC subnets).

Supported Engines

RDS Proxy supports MySQL, PostgreSQL, MariaDB, and SQL Server (Aurora MySQL and Aurora PostgreSQL too).

Exam Patterns

  • "Lambda + RDS + connection issues / TooManyConnections" → RDS Proxy.
  • "Reduce database failover time for Aurora" → RDS Proxy.
  • "Lambda needs to securely access RDS without storing credentials" → RDS Proxy + IAM auth + Secrets Manager.
  • "Serverless application connecting to relational database at scale" → RDS Proxy.
  • RDS Proxy is NOT needed for DynamoDB (it's only for relational DBs). If the question mentions DynamoDB → RDS Proxy is a distractor.
  • RDS Proxy does NOT support RDS for Oracle at the time of exam scope.

When NOT RDS Proxy

If the workload is read-heavy → consider Aurora Read Replicas. If traffic is unpredictable + NoSQL is acceptable → consider DynamoDB. RDS Proxy solves the connection problem specifically, not performance or scaling of the DB itself.

Section 4 Quiz — RDS Proxy

Q1. A company's serverless application uses Lambda functions to write to an Aurora MySQL database. During peak hours, they see TooManyConnections errors and the database becomes unresponsive. What should the solutions architect recommend?

Lambda + Aurora + TooManyConnections = RDS Proxy. Increasing instance size raises the connection limit marginally but doesn't fix the fundamental problem. Reducing Lambda concurrency limits throughput. Switching to DynamoDB is a redesign, not a solution to the stated problem.

Q2. A solutions architect wants Lambda functions to authenticate to RDS without hardcoded database credentials. The solution should follow AWS security best practices. Which approach is correct?

Best practice: RDS Proxy + IAM authentication. Lambda authenticates to Proxy via IAM role (no credentials in code). Proxy gets DB credentials from Secrets Manager with auto-rotation. A and B store credentials, which is less ideal. D is a terrible pattern.

Q3. A company uses Aurora PostgreSQL with Multi-AZ. During a failover event, their application experiences 30+ seconds of downtime. They want to reduce failover impact. What should they implement?

RDS Proxy reduces failover time by up to 66% by maintaining connection pools and transparently routing to the new primary. The application doesn't need to re-establish connections. Aurora Global Database is for cross-region DR, not in-region failover improvement. ElastiCache doesn't help with failover. Serverless v2 is about scaling, not failover speed.

Q4. A Lambda function connects to an RDS MySQL instance through RDS Proxy. Where must the RDS Proxy be deployed?

RDS Proxy must be in the same VPC as the target RDS/Aurora database. It can't be in a separate VPC. It's not a public internet endpoint. Lambda must also be VPC-configured to reach the proxy.

Q5. A serverless application uses Lambda with API Gateway. The backend is a PostgreSQL database. The team needs to handle sudden spikes from 100 to 10,000 concurrent requests while keeping database connections manageable. The solution must minimize operational overhead. Which architecture is best?

Lambda scales automatically for the spike. RDS Proxy handles connection pooling. Aurora PostgreSQL is managed. This ticks all boxes: serverless, handles spikes, manages connections, low ops overhead. ElastiCache (A) doesn't solve write connection problems. ECS fixed tasks (B) won't handle the spike. Migration to DynamoDB (D) is a rewrite.

5. Cost Optimization Tools

Multiple AWS services help with cost management. The exam tests which tool to use for which situation.

AWS Cost Explorer

  • Visualize, understand, and manage AWS costs and usage over time.
  • Shows up to 12 months of historical data.
  • Can forecast future costs (up to 12 months ahead).
  • Granularity: monthly, daily, or hourly (hourly only for last 14 days).
  • Built-in Reserved Instance / Savings Plan recommendations.
  • Filter and group by service, account, tag, region, etc.
  • Exam keyword: "analyze spending trends" or "forecast costs" → Cost Explorer.

AWS Budgets

  • Set custom budgets and get alerts when costs/usage exceed thresholds.
  • 4 budget types: Cost, Usage, Reservation, Savings Plans.
  • Can trigger actions: apply IAM policy, apply SCP, stop EC2 instances.
  • Supports SNS notifications and Chatbot integration.
  • Exam keyword: "alert when spending exceeds $X" or "automate response to overspending" → AWS Budgets.

AWS Cost and Usage Report (CUR)

  • The MOST detailed cost dataset. Line-item granularity.
  • Delivered to S3 as CSV/Parquet. Can be queried with Athena, Redshift, or QuickSight.
  • Includes resource-level IDs (which specific EC2 instance, which S3 bucket).
  • Exam keyword: "most detailed billing data" or "resource-level cost breakdown" or "query billing data with Athena" → CUR.

AWS Trusted Advisor

  • Automated best practice checks across 5 categories: Cost Optimization, Performance, Security, Fault Tolerance, Service Limits.
  • Free tier (Basic/Developer): Only core security checks + service limits.
  • Business/Enterprise Support: Full set of checks including cost optimization.
  • Cost examples: idle EC2 instances, underutilized EBS volumes, unassociated Elastic IPs, idle RDS instances.
  • Exam keyword: "identify idle/underutilized resources" or "best practice recommendations" → Trusted Advisor.

AWS Compute Optimizer

  • Uses ML to analyze utilization metrics and recommend right-sizing for: EC2 instances, EBS volumes, Lambda functions, ECS on Fargate.
  • Tells you if instances are over-provisioned or under-provisioned.
  • Exam keyword: "right-size EC2 instances" or "optimal instance type" → Compute Optimizer.

Savings Plans & Reserved Instances

  • Savings Plans: Commit to a consistent amount of usage ($/hr) for 1 or 3 years. More flexible than RIs. Two types: Compute Savings Plans (any region, any instance family, any OS) and EC2 Instance Savings Plans (specific region + instance family).
  • Reserved Instances: Commit to specific instance type in a specific region. Standard RIs (up to 72% off) or Convertible RIs (up to 66% off, can change instance family).
  • Exam keyword: "reduce EC2 costs for steady-state workloads" → Savings Plans or RIs. If they want flexibility → Savings Plans. If they want maximum discount + know the instance type → Standard RI.

S3 Cost Optimization

  • S3 Intelligent-Tiering: Automatically moves objects between access tiers based on usage. No retrieval fees. Small monthly monitoring fee. Best for unknown/changing access patterns.
  • S3 Lifecycle Policies: Rule-based transition (e.g., move to IA after 30 days, Glacier after 90, delete after 365).
  • S3 Storage Lens: Org-wide visibility into S3 usage and activity trends. Identifies cost optimization opportunities.
  • Exam pattern: "unknown access patterns" → Intelligent-Tiering. "Known pattern, automate transitions" → Lifecycle policies.

Section 5 Quiz — Cost Optimization

Q1. A company wants to receive an email notification whenever their monthly AWS spend exceeds $10,000. They also want to automatically restrict non-production IAM users from launching new EC2 instances when spending exceeds $15,000. Which service should they use?

AWS Budgets supports alerts AND automated actions (apply IAM policies, SCPs, stop instances). Budget actions can restrict IAM permissions when thresholds are breached. Cost Explorer doesn't have built-in actions. Trusted Advisor doesn't do budget alerts. CUR + Lambda is over-engineering.

Q2. A finance team needs to query their AWS billing data at the individual resource level using Amazon Athena. They want to identify which specific EC2 instances and S3 buckets are the most expensive. Which tool provides this data?

CUR provides the most granular billing data with resource-level IDs. It's delivered to S3 as Parquet/CSV and can be queried with Athena. Cost Explorer doesn't support Athena queries and has less granularity. Budgets is for alerts, not analysis. Trusted Advisor doesn't provide billing line items.

Q3. A company runs a fleet of EC2 instances with varying utilization. Some instances are consistently at 5% CPU utilization while others run at 90%. They want an automated recommendation on which instances to downsize or terminate. Which service should they use?

Compute Optimizer uses ML to analyze CloudWatch metrics and recommends optimal instance types (right-sizing). It tells you exactly which instances are over/under-provisioned and what to change to. Trusted Advisor flags idle instances but doesn't recommend specific right-sizing. Cost Explorer shows spend but not instance-level recommendations. CloudWatch shows metrics but doesn't recommend actions.

Q4. A solutions architect wants to reduce EC2 costs for a workload that runs 24/7 across multiple regions, and may need to change instance families as requirements evolve. Which purchasing option provides the best balance of savings and flexibility?

Compute Savings Plans offer the most flexibility — any region, any instance family, any OS, even Fargate and Lambda. EC2 Instance Savings Plans are locked to a region + instance family. Standard RIs are the most rigid. Convertible RIs allow changing but give less discount than Savings Plans and are more operationally complex.

Q5. A company stores 50 TB of data in S3 Standard. Some objects are accessed frequently for the first month, then rarely. Some objects are never accessed after upload. They want automatic cost optimization without managing lifecycle rules. Which storage class should they use?

"Automatic" + "without managing lifecycle rules" + varying access patterns = S3 Intelligent-Tiering. It automatically moves objects between frequent and infrequent access tiers (and archive tiers if configured) based on actual usage. No retrieval fees, just a small monitoring fee per object. Lifecycle policies (A) require manual rule management which they want to avoid.

Q6. A company with AWS Enterprise Support wants to identify unassociated Elastic IP addresses, idle RDS instances, and underutilized EBS volumes across their accounts. Which service provides these checks out of the box?

Trusted Advisor with Business/Enterprise support provides cost optimization checks including: idle ELBs, unassociated EIPs, underutilized EBS, idle RDS instances, and more. Compute Optimizer is for right-sizing compute. Cost Explorer shows spend trends. Inspector is for security vulnerabilities.
